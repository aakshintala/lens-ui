Problem

Absolute tile geometry is computed inside gpui render closures, so the only way to verify it is a real window — and three other modules re-derive it by hand.

Solution

pack returns fully-resolved tile rects (group box, header band, member cells, cull flag, centering offset); the render methods become painters over rects they no longer compute.

Wins
  • Locality: one formula, one module
  • Leverage: painter and hit-test share rects
  • Culling and centering testable pure
  • B-4c drag stops copying geometry
  • Interface shrinks; implementation absorbs 500 render lines
  • Deletes the doc-comment invariant
Load-bearing for the next build. The B-4c spike (HEAD commit, docs/spikes/2026-07-23-board-b4c-drag-reverse-hittest.md) resolves a drop by scanning the placed tiles “which the renderer already computes” — but they only exist inside the renderer, so the spike rebuilt them. Cross-family review then found a real off-by-one in that copy.

move_item* = the 13th method B-4c is about to add.

Problem

The BoardStore interface is shallow — twelve methods that each mirror one BoardLayout method and repeat the same transaction envelope. Every new board mutation costs a trait method, an impl, a replica Op variant, and a match arm.

Solution

Move replica::Op down into lens-core as BoardOp; the trait becomes mode + load_layout + apply(&BoardOp). The envelope is written once and the op enum becomes the seam.

Wins
  • Interface 12 → 3
  • Envelope written once, not twelve times
  • Locality: transaction bugs concentrate
  • An in-memory adapter becomes cheap
  • B-4c adds a variant, not a method
  • Op enum is the test surface
Seam check. Four store traits — ControlStore, TranscriptStore, TranscriptReader, BoardStore — have four sqlite adapters and zero fakes; every test opens a real tempfile db. One adapter means a hypothetical seam. Either narrow the interfaces until a second adapter is cheap (this card), or drop the traits and name the structs directly.

Problem

770 lines where the transition decision, the teardown side effect, and the gpui notification are one method taking &mut Context<Self>. No transition is verifiable without gpui, and the guards are per-method rather than tabular.

Solution

The crate already has the shape: generation.rs is a pure reducer returning a GenerationVerdict. Extend it to the whole lifecycle — events in, (Lifecycle, Vec<Effect>) out — and let TerminalTab be the effect executor.

Wins
  • Transition table is the test surface
  • Late-event gating: one arm, not N guards
  • Locality: epoch bumps in one place
  • Whole-branch-only bugs become unit-visible
  • lib.rs sheds ~770 orchestration lines
  • Leverage: same reducer drives the demo host
Evidence this is real. The Slice-4 whole-branch review caught a Critical the per-task reviews “structurally couldn’t see”: apply_bridge_event had no lifecycle gate, so a late 4404 close clobbered ReplacementWaiting/Sleeping (fixed in f5ced39). A transition table makes that class of gap a missing match arm.

Problem

ShellMode lives inside board/mod.rs, and BoardView renders the nav rail, titlebar, replica banner, focused rail and chat tab as well as the board. Its interface carries eight pub fn *_for_test hatches — tests reach past the interface rather than through it.

Solution

A ShellView module owns mode, chrome and the content slot; BoardSurface renders tiles only. The repo's own design doc already draws this line — the shell owns “containers and chrome”, surfaces own “content that fills the slots”.

Wins
  • Shell changes stop touching tile geometry
  • Escape hatches collapse toward one interface
  • Leverage: Slice 6 docks, doesn't edit board
  • Locality: mode transitions in one module
  • Code matches the design doc's split
Don't design the tab seam yet. pub trait ContentTab {} (slot/mod.rs:7) has no methods and one real adapter — a hypothetical seam. lens-ui doesn't depend on lens-terminal at all today. Terminal Slice 6 is the second adapter; that's when the interface can be designed from two concrete sides rather than one (memory: premature-layer-boundary-binding).

Problem

~580 impl lines fuse the live/settled split, finalize staging, expansion flags, marker placement and disk-read merge into one state machine where every method takes &mut Context<Self>. The pure decisions can't be reached without gpui, so 35 #[gpui::test]s plus three real-window probe binaries carry the load.

Solution

Split decide from apply. A pure plan() over borrowed state returns what should change; the replica applies it. The precedent is one crate away — lens-core/reduce/view.rs is already a pure borrow-only projection.

Wins
  • Plan is the test surface, not the window
  • Probes shrink to what only a window proves
  • T-2b windowing lands against a pure interface
  • Locality: read-range policy in one place
  • Fewer hand-authored probe states
Why the probes are the tell. All three hand-author state production owns — new_test_no_baseline, ReaderWorkerHandle::new_test(), hardcoded generation: 1. This repo already has the scar: memory false-green-probe-drives-production-path records a probe that validated the harness rather than the product, and the T-2 review found it had falsely validated the crux.

Contradicts a locked design decision — and should stay that way. capability-map-and-design-language.md §0.1.3 commits to “omnigent-native, one seam”: no Backend trait, no normalized cross-backend contract, the domain model is the omnigent model. ADR-0001 localizes contract churn to the vendored pin. With one server and one contract, a translation stage would be a shallow module: deleting it moves complexity rather than concentrating it.
The observation

This is the largest coupling in the repo — the reducer pattern-matches SessionEvent, ResponseEvent and WireItem across four modules. lens-ui, by contrast, is clean: it sees only ActorFeed / StreamUpdate.

Recorded so it isn't re-derived

Listed here only so the next architecture pass doesn't surface it as a finding. Revisit if — and only if — Lens ever targets a second server contract. The cheap hedge meanwhile is the existing drift check, not an abstraction.

Top recommendation

1 · Resolve the board frame in pack

It is the only candidate whose payoff is already scheduled. B-4c drag needs the placed tiles the renderer computes; because they live inside gpui closures, the spike rebuilt the geometry a third time and review found a real off-by-one in the copy. Doing this first turns B-4c's reverse hit-test from a parallel implementation into a read of the same rects the painter uses — and pays down four copies of one formula on the way.

Natural follow-on: candidate 2, since B-4c also wants a thirteenth BoardStore method. Doing both before the build means B-4c adds one enum variant and one match arm instead of a method, an impl, an op and a dispatch arm.
